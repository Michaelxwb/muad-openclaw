## 1. 引言

### 1.1 目的

本文档给出 **muad-openclaw** 当前版本的系统级总体设计，供技术经理、架构师、研发与运维在 TMG 评审与概要设计前对齐：

| 项 | 说明 |
| -- | ---- |
| **输入** | 企微私聊交付、SOP Skill 化、约 100 用户、不 fork OpenClaw；**业务场景需快速验证**与**生产 SOP 需可靠执行**双诉求 |
| **输出** | 模块划分、**双执行路径边界**、关键系统流程、部署与容量基线、DFX 目标与手段 |
| **读者** | 项目技术经理、架构师、研发、测试、运维、安全经理 |

**总设责任边界（模板要求）**：需求侧已明确的能力，须在本总设中有架构体现；回溯时若需求有而总设无体现，责任在总设。

### 1.2 适用范围和评审要求

#### 1.2.1 适用范围

| 场景（模板） | 本版本决策 |
| ------------ | ---------- |
| 架构大面积调整 | **是**：多用户单 Pod + 控制面；并明确 **脚本快速验证与工作流可靠执行双路径并存** |
| 决策 | **做一份全新总体设计**（相对早期单用户验证形态） |

**本总设覆盖范围：**

1. 多用户 Runtime（Pod / Agent / workspace / browser / 模型）与控制面 Console  
2. 企微/微信身份与绑定、Runtime 配置事务 apply  
3. **脚本型路径**：Runtime 内 Skill 工具链（隔离、执行、凭证、进度、并发）— 业务场景快速验证  
4. **工作流型路径**：自动化平台（工作流引擎 + 能力中心 + 可选浏览器）+ Bridge Skill — 可靠 SOP 执行  
5. 两条路径与 Console / Runtime 的协作、并存策略与选型边界  
6. 约 100 用户容量规划摘要（分路径）

**不做强承诺 / 排除：** 全局 Message Router；控制面本阶段强制分布式数据库/缓存/向量库；群聊强隔离；多用户共享同一微信登录态；browser 上游自动选 profile 的无条件保证；专家知识库产品化（可后置）。

#### 1.2.2 评审要求

总体设计评审统一提交电子流，由 TMO 组织 **TMG** 评审（模板默认：可靠性 TMG 参加）。  
**待填**：电子流单号、评审日期、与会人。

### 1.3 定义和缩写

| 缩写/名 | 描述 | 备注 |
| ------- | ---- | ---- |
| Console | 控制面（管理 API + 管理 UI） | 管理面唯一入口 |
| Runtime Pod | 单 OpenClaw Gateway 运行单元 | 多 Human User 共享；对话与触发中枢 |
| Human User | 平台内部用户实体 | 不等于 IM 外部账号 |
| Identity | IM 身份到 Human User 的绑定 | channel + peer |
| Runtime 配置 | 版本化运行时配置（generation） | apply 事务单元 |
| **脚本型路径** | 在 Pod 内以脚本 Skill 执行 SOP | 适合快速验证、短链路 |
| **工作流型路径** | Bridge + 自动化平台工作流执行 SOP | 适合定型生产、可恢复 |
| 工具链（脚本路径） | 门禁 / Skill 执行器 / 会话管理 / 进度 / 适配 / 并发 | 脚本型主路径组件 |
| Bridge Skill | Agent 触发 SOP 的入口契约 | 语义 + HTTP 调用，不承载重脚本 |
| 自动化平台 | 工作流引擎 + Worker + 能力中心 | 工作流型执行面 |
| 能力中心 | 版本化业务 API/浏览器能力封装 | 禁止能力间互调；组合在工作流层 |
| grant | Skill 授权清单 | 用户生效视图 |
| apply | 准备→校验→提交→健康/回滚 | 配置下发事务 |
| SOP | 标准服务操作流程 | 可走脚本路径或工作流路径 |
| MSSW / SDSP | 业务平台（产品范围） | Skill / 能力中心调用 |
| LLM | 大语言模型服务 | 模型池按用户绑定 |

### 1.5 总设任务书

| 编号 | 目标项概述 | 对应的标准要求（初稿） |
| ---- | ---------- | ---------------------- |
| 1 | 性能参数要求 | **规划值**：约 100 服务经理并行。**脚本路径**：默认约 10 用户/Pod，Skill/Browser 并发可配（浏览器为瓶颈）。**工作流路径**：目标提高单 Pod 用户密度（对话容器轻量化，需压测）；工作流并发由任务队列与 Activity 并发控制。LLM 时延不在本平台 SLA 内。 |
| 2 | 资源开销要求 | **规划值**：以脚本路径为主时约 10 Runtime Pod + 浏览器资源，集群量级约 40–48C / 96–128Gi / 500Gi+。启用工作流路径后叠加自动化平台、工作流服务与执行库；Runtime 侧重对话，浏览器可后置到自动化平台；合计以压测为准。Console 约 2C/4G。 |
| 3 | 可靠性 | **共用**：配置 apply 可回滚；Console 故障时已绑定对话通常可继续。**脚本路径**：无强任务持久化，接受「快速验证」级可靠性。**工作流路径**：重试、补偿、断点续传与事件历史；写操作须同步设计补偿。 |
| 4 | 架构要求 | 控制面与运行时分离；不 fork OpenClaw；**双路径并存且可按 SOP 选型**；用户/模型/Skill/凭证隔离；无全局 Message Router。 |
| 5 | 可运维性 | Console 统一开通与配置；自动化平台组件可独立部署与扩缩；日志与执行记录可分路径查询。 |
| 6 | 可测试性 | 控制面可用模拟编排驱动测试；脚本工具链单测；工作流可对瞬时故障/补偿/恢复做场景验证。 |
| 7 | 可扩展性 | 水平加 Pod；**新场景优先脚本 Skill 快速试跑**；**定型 SOP 优先复用能力中心、新增工作流定义**；通道可扩展。 |
| 8 | 可调试性 | apply/门禁/执行记录可观测；工作流路径另具备事件历史与执行存储查询。 |
| 9 | 兼容性 | 固定 OpenClaw 镜像代际；同代 Runtime 配置兼容；脚本 Skill 与 Bridge Skill 可同集群共存，互不强制迁移。 |
| 10 | 安全性 | 密钥不进镜像与普通日志；每用户隔离；自动化平台鉴权（令牌/后续 mTLS）；安全兜底清单待附件。 |
| 11 | 可重用性 | 产出：多用户控制面、脚本工具链模式、能力中心 + 工作流模板。 |
| 12 | 其它要求 | 业务平台范围 **MSSW、SDSP**；**脚本路径用于快速验证，不替代工作流路径的生产级可靠执行承诺**。 |

### 2.4 关键技术及解决途径

| 关键问题 | 途径 |
| -------- | ---- |
| 不改上游如何扩展 | 外置插件/CLI + 版本化 Runtime 配置 |
| 多用户如何隔离 | agent / workspace / profile + 门禁 |
| **场景如何快速验证** | **脚本型 Skill**，少组件、短闭环 |

| Pod 不被打满 | 脚本：并发租约；工作流：重 IO/浏览器可迁出 Pod |
| 两路径如何共存 | 同一 Agent 按 Skill 类型选择入口；控制面统一授权与审计视角 |

## 3. 硬件方案

**不涉及。** LLM 在外部模型服务完成。

## 4. 软件方案

### 4.1 方案选型

#### 4.1.1 决策点一：Agent 内核（OpenClaw vs Hermes）

| 评估准则 | 权重 | OpenClaw 加权 | Hermes 加权 |
| -------- | ---- | ------------- | ----------- |
| 企微私聊交付体验 | 30% | 1.50 | 0.90 |
| 企业 Skill 分层 | 25% | 1.25 | 0.50 |
| 跨 IM 身份关联 | 15% | 0.75 | 0.30 |
| 多用户会话工程化 | 15% | 0.60 | 0.75 |
| 不 fork 扩展成本 | 10% | 0.50 | 0.40 |
| 群聊 per-user（低权重） | 5% | 0.15 | 0.25 |
| **合计** | 100% | **4.75** | **3.10** |

| 方案 | 优点 | 风险 | 选择 |
| ---- | ---- | ---- | ---- |
| **OpenClaw** | 主入口匹配；Skill 分层与跨 IM 关联贴产品 | 多用户底座自建 | **是** |
| Hermes | 会话/群聊强 | 跨 IM 与 Skill 分层补齐成本高 | 否（对照） |

#### 4.1.2 决策点二：部署单元

| 方案 | 结论 |
| ---- | ---- |
| 每用户一 Pod | 否（稳态成本） |
| **单 Pod 多用户约 10（脚本路径默认）** | **是**；工作流路径下目标提高对话密度（压测后） |
| 全局 Message Router | 否（当前） |

#### 4.1.3 决策点三：SOP 执行架构（**双路径并存**）

| 评估准则 | 权重 | 仅脚本路径 | 加权 | 仅工作流路径 | 加权 | **双路径并存** | 加权 |
| -------- | ---- | ---------- | ---- | ------------ | ---- | -------------- | ---- |
| 新场景上线速度 | 30% | 5 | 1.50 | 2 | 0.60 | 5 | 1.50 |
| 长任务可恢复/可补偿 | 25% | 2 | 0.50 | 5 | 1.25 | 5 | 1.25 |
| 能力复用与审计 | 20% | 2 | 0.40 | 5 | 1.00 | 5 | 1.00 |
| 运行时资源（浏览器等） | 15% | 2 | 0.30 | 5 | 0.75 | 4 | 0.60 |
| 工程与运维复杂度 | 10% | 5 | 0.50 | 3 | 0.30 | 3 | 0.30 |
| **合计** | | | **3.20** | | **3.90** | | **4.65** |

| 方案 | 优点 | 风险 | 选择 |
| ---- | ---- | ---- | ---- |
| 仅脚本路径 | 最简单，验证最快 | 生产长任务可靠性不足 | 否（单独作稳态） |
| 仅工作流路径 | 可靠、可复用 | 每个试错场景都上工作流，拖慢验证 | 否（扼杀速度） |
| **双路径并存** | **验证走脚本，生产可靠走工作流** | 双路径治理与 Skill 分类 | **是** |

**并存原则：**

1. **同一对话底座**（Console + Runtime + 门禁 + 模型/绑定）共用。  
2. **按 SOP/Skill 选型**：快速验证、接口联调、演示 → 脚本型；多步写操作、长时、需恢复与强审计 → 工作流型。  
3. **不强制一刀切迁移**：成熟脚本 Skill 可逐步改为 Bridge + 工作流；未稳定场景保留脚本。  
4. **用户无感路径差异**：服务经理仍企微一句话；差异在 Skill 实现形态与后端执行面。

#### 4.1.4 选型结论（稳态）

| 决策点 | 选择 |
| ------ | ---- |
| Agent 内核 | OpenClaw |
| 部署单元 | 单 Pod 多用户 |
| 扩展方式 | 不 fork；外置工具链 + 自动化平台 |
| **SOP 执行** | **脚本型路径 ∥ 工作流型路径 并存** |
| 消息路由 | 管理员分配 Pod + 绑定路由 |

### 4.3 系统流程

#### 4.3.1 流程1：开通与配置（共用）

**【图 4-8 开通与配置时序 — 待补充】**  
参与者：管理员、Console、编排驱动、Runtime、（可选）自动化平台配置注入。

2. 预录 Identity 或绑定码

#### 4.3.2 流程2：绑定激活（共用）

**【图 4-9 绑定激活时序 — 待补充】**  
参与者：服务经理、IM、Gateway、门禁、Console、apply。

门禁 → 绑定激活 → 生成 agent/workspace/profile → apply 更新路由。

#### 4.3.3 流程3a：脚本型快速验证 SOP（USE-05a）

**【图 4-10 脚本型 SOP 时序 — 待补充】**  
底稿：下列 ASCII 链路，请画为 UML 顺序图（含进度旁路与执行上报）。

```text
IM → Gateway → 门禁 → Agent
  → Skill 执行器 → 脚本
       → 进度组件 → 适配 → IM 进度
       → 会话管理 → Console 凭证解析
       → 并发控制
  → 原生终态回复
  → 执行上报（失败可缓存重试）
```

**适用：** 新接口联调、演示、步骤少、可接受失败重跑的场景。  
**不适用作为唯一手段：** 长时多步写操作、必须断点续传与强补偿。

#### 4.3.4 流程3b：工作流型可靠 SOP（USE-05b）

**【图 4-11 工作流型 SOP 时序 — 待补充】**  
底稿：下列 ASCII 链路，请画为 UML 顺序图（含重试/补偿注记）。

```text
IM → Gateway → 门禁 → Agent
  → Bridge Skill（读契约）
  → HTTP 触发自动化平台
       → 工作流
            → Activity 调用能力中心（重试策略）
            → 写操作配套补偿 Activity
            → 可选浏览器 Activity（用后销毁）
  → 查询/回调结果 → Agent 解释 → IM 终态
```

**容错基线（工作流路径）：**

| 故障类型 | 策略 |
| -------- | ---- |
| 瞬时故障 | 指数退避重试（如 1s 起，上限与次数受限） |
| 业务预期失败 | 不重试，工作流失败 |
| 部分成功需回滚 | Saga 补偿；写操作必须同步设计补偿 |

#### 4.3.5 流程4：路径选型（治理）

| 条件 | 选择 |
| ---- | ---- |
| 场景未定型、需一周内联调见效 | 脚本型路径 |
| 已定型、步骤多、有写操作与通知/发布 | 工作流型路径 |
| 浏览器导出重、拖垮对话 Pod | 优先工作流型（浏览器后置） |
| 仅对话/轻工具、无多步业务 API | 可无 SOP 执行面，或极简脚本 Skill |

| 顺序 | 初始化 | 退出 |
| ---- | ------ | ---- |
| 1 | 基础设施就绪 | 停新用户分配 |
| 2 | Console 启动 | 排空脚本任务 / 等待工作流策略（取消或续跑，待运维定义） |
| 3 | Runtime 拉起 + 配置注入 | 优雅停 Gateway |

信任边界：管理员凭证 ≠ Pod 令牌 ≠ IM 身份 ≠ 自动化平台令牌。  
脚本与工作流路径均不得将长期业务密钥写入 Skill 正文。

#### 5.1.4 预使用组件版本安全情况

Mend 扫描；主要组件含 OpenClaw、控制面依赖、工作流与自动化平台依赖、浏览器组件。  
结果 **待填**。

### 5.2 可靠性设计

#### 5.2.1 能力目标定义

| 一级目标 | 脚本型路径（快速验证） | 工作流型路径（可靠执行） |
| -------- | ---------------------- | ------------------------ |
| 配置可靠 | 共用：apply 可回滚 | 同左 |
| 故障半径 | 单 Pod 约 10 用户（默认） | 对话 Pod 半径 + 工作流任务级隔离 |
| 任务续跑 | **不承诺** Pod 重启后续跑 | **承诺**工作流级续跑（引擎能力） |
| 写操作补偿 | 脚本自管，弱 | **强制**补偿设计 |
| 可观测 | 执行上报+日志 | 事件历史+执行存储 |

#### 5.2.2 故障场景分析

| 场景 | 脚本型路径 | 工作流型路径 |
| ---- | ---------- | ------------ |
| Console 宕机 | 已绑定对话通常可；开通/解析降级 | 同左；进行中工作流取决于是否依赖 Console |
| Runtime 宕机 | 该 Pod 用户与进行中脚本任务失败 | 对话中断；**已提交工作流可由 Worker 续跑** |
| 脚本中途失败 | 常需整单重来 | Activity 级重试/补偿 |
| 自动化平台宕机 | 不影响纯脚本路径 | 生产工作流 SOP 不可新触发；已有工作流策略依引擎 |

#### 5.2.3 业界竞争分析

| 维度 | 常见 | 本方案 |
| ---- | ---- | ------ |
| 验证速度 | 脚本/低代码 | **保留脚本闭环** |
| 生产可靠 | 工作流引擎 | **自动化平台 + 成熟工作流引擎** |
| 多租户 Agent | 独立容器或逻辑隔离 | 单 Pod 多用户 + 门禁 |

#### 5.2.4 实现方案设计

**【图 5-2 故障影响与冗余 — 待补充】**  
建议内容：Console / 单 Runtime / 自动化平台 三类故障时，对话、新绑定、脚本任务、已提交工作流各自是否可继续（与 §5.2.2 表一致，图形化）。

**冗余：** Runtime 多 Pod 分片；自动化平台/Worker 可多副本；Console 本阶段单实例+备份。  

**故障管理：** 容器探针重启；apply 回滚；脚本路径上报缓存；工作流引擎级重试与补偿。  

**升级：** Runtime 滚动；自动化平台组件独立滚动；工作流定义版本兼容策略待概要细化。  

**应急：** 控制面备份恢复；工作流执行历史用于定责与人工补偿。

#### 5.2.5 韧性设计

**【图 5-3 过载保护与限流点 — 待补充（可选）】**  
建议内容：Pod 内并发租约、工作流任务队列、LLM 独立密钥、管理面与数据面分离。

| 过载 | 策略 |
| ---- | ---- |
| 脚本 Skill/浏览器并发 | 租约上限，拒绝或排队 |
| 工作流 Activity 并发 | 任务队列与 Worker 并发限制 |
| LLM 限流 | 单用户密钥隔离影响面 |
| 平台攻击面 | 超时与无效请求抑制（部署与网关层） |

## 6. 总结

### 6.1 与上一个版本的设计变化

#### 6.1.1 单 Pod 多用户 + Console

相对每用户一 Pod：成本可控，故障半径显式化，平台化治理。

#### 6.1.2 外置扩展不 fork

门禁与工具链/自动化平台均在上游之外扩展。

#### 6.1.3 **脚本快速验证与工作流可靠执行并存（本版关键）**

| 项 | 内容 |
| -- | ---- |
| 变化原因 | 仅脚本无法满足生产可靠；仅工作流拖慢业务验证 |
| 变化描述 | 脚本路径保留并明确「快速验证」定位；工作流可靠执行层并列交付 |
| 影响 | Skill 分类治理、双部署单元、可靠性指标分路径承诺 |

### 6.2 风险问题保证

| 风险点 | 规避措施 |
| ------ | -------- |
| 双路径认知混乱 | 总设与运营规范明确「何时用脚本/工作流」；Skill 元数据标识类型 |
| 用脚本路径扛生产长任务 | 评审与上线清单：长链路强制工作流路径 |
| 用工作流做所有试错 | 允许脚本快速验证，避免工作流爆炸 |
| 逻辑隔离 | 门禁 + 用户上限 + 验证 |
| 控制面单点 | 备份；规模再评估 |
| 自动化平台依赖 | 多副本、监控、降级策略（仅对话/脚本仍可用） |
| 能力对接不齐 | 真实 API 收口清单 |
| 群聊/微信共享登录 | 不做强承诺 |

### 6.3 尚未解决的问题

| 问题 | 计划 |
| ---- | ---- |
| 脚本→工作流迁移判据与清单 | 产品+架构固化 |
| 工作流真实 API / 浏览器能力收口 | 实现与联调 |
| 双路径统一审计视图 | 产品设计 |
| 压测基线（分路径资源） | 性能测试 |
| 安全附件与 Mend | 安全经理 |
| 型号代号、电子流 | 项目管理 |
