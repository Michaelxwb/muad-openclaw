## 2. 背景与目标

### 2.1 问题陈述

| 维度 | 内容 |
|------|------|
| **问题描述** | 当前平台主要实现了 K8S 相关后端能力，缺少对 agent 使用场景的架构级观测能力。平台后续需要了解不同用户、不同 agent、不同能力调用链路的使用情况，但 agent 底座不固定，不能把埋点实现绑定到某一个 agent 代码结构。 |
| **当前替代方案** | 直接改某个 agent 的核心代码、扫日志、或者为每个 agent 单独写非标准采集逻辑。上述方式接入成本高、可复用性弱、隐私边界不清晰，且难以支持后续更换 agent 底座。 |
| **触发原因** | 需要为 muad-openclaw 增加长期可演进的 agent 使用观测能力。前期已评估 OpenClaw、Hermes Agent、GenericAgent，其中 OpenClaw 与 Hermes Agent 作为首批重点适配对象，GenericAgent 作为后续压力测试参考。 |

---

### 2.2 目标与价值

| 维度 | 内容 |
|------|------|
| **核心目标** | 建立一套 agent 无关、插件化、可插拔的遥测接入架构，让不同 agent runtime 可以通过适配器低成本接入统一事件协议。 |
| **预期价值** | 降低新 agent 接入成本；避免平台与单一 agent 强耦合；为后续用户行为分析、skill 调用分析、工具/LLM 使用分析、运维审计与看板展示提供稳定数据基础。 |

**成功指标**

| 指标 | 当前值 | 目标值 | 衡量方式 |
|------|--------|--------|----------|
| agent 接入可行性 | 缺少统一评估标准 | 每个候选 agent 都能按同一套清单判断是否可接入、需要何种改造 | 设计阶段产出 adapter capability matrix |
| 首批适配覆盖 | 未形成统一适配方案 | OpenClaw 与 Hermes Agent 均有明确 adapter 方案与事件映射 | 设计评审确认 OpenClaw/Hermes 映射表 |
| 接入方式 | 依赖定制改造或日志推断 | 优先通过插件或 hook 接入，核心代码改动仅限通用扩展点 | 设计中标注每个事件来源与侵入等级 |
| 指标口径 | 暂未定义 | 本期不定义具体业务指标，仅保证事件架构可承载后续指标 | 范围评审确认具体指标延后 |

## 4. 功能需求

### 4.1 功能清单

| 功能ID | 功能名称 | 功能描述 | 优先级 | 来源用户故事 |
|--------|---------|---------|--------|-------------|
| FEAT-01 | 统一 agent 遥测事件抽象 | 定义 runtime 无关的事件模型、上下文模型和事件分类，支撑不同 agent 输出同一种平台可消费事件。 | P0 | US-01, US-06 |
| FEAT-02 | 插件化适配器架构 | 定义 adapter 插件接口、加载/启停方式、能力声明、事件上报契约和失败隔离规则。 | P0 | US-01, US-03, US-04 |
| FEAT-03 | OpenClaw 首批适配方案 | 细化 OpenClaw 的可采集事件来源、插件/hook 接入点、缺口与最小改造建议。 | P0 | US-02, US-04 |
| FEAT-04 | Hermes Agent 首批适配方案 | 细化 Hermes Agent 的 hooks、observability 插件模式、skill 采集语义和 adapter 事件映射。 | P0 | US-02, US-04 |
| FEAT-05 | agent 接入评估机制 | 提供候选 agent 的接入评估清单，包括入口消息、会话、LLM、tool、skill、用户身份、权限与隐私能力。 | P0 | US-04 |
| FEAT-06 | 统一 collector/exporter 边界 | 定义 adapter 与平台侧 collector 的职责分界，支持本地缓冲、统一上报、失败重试和后续多 exporter 扩展。 | P0 | US-03, US-06 |
| FEAT-07 | 隐私、脱敏与显式启用策略 | 要求所有遥测插件默认关闭或显式配置启用，并定义用户内容、工具参数、secret、用户 ID 的采集/脱敏策略。 | P0 | US-05 |
| FEAT-08 | 新 agent 快速接入模板 | 为后续 agent 提供 adapter 模板、事件映射模板和接入说明，降低重复适配成本。 | P1 | US-03, US-04 |
| FEAT-09 | 指标与看板延后承载约束 | 明确本期只做架构层事件承载，不固化具体指标口径，但事件模型需能支持后续扩展。 | P1 | US-06 |

#### FEAT-01: 统一 agent 遥测事件抽象

- **描述**: 定义统一事件协议，例如 `agent-telemetry-event.v1`，让不同 agent adapter 输出一致结构。
- **验收标准**:
  - [ ] 事件模型包含 runtime、session、turn、user、message、tool、LLM、skill/能力、error 等基础上下文。
  - [ ] 事件模型不依赖 OpenClaw、Hermes 或任意单一 agent 的内部类型。
  - [ ] 事件模型允许后续新增事件类型且保持版本兼容。
  - [ ] 明确“事件采集”与“指标计算”边界，本期不要求定义最终业务指标。

#### FEAT-02: 插件化适配器架构

- **描述**: adapter 必须以插件化方式接入 agent runtime，平台通过统一接口加载、配置和管理。
- **验收标准**:
  - [ ] adapter 能声明自身支持的事件能力与限制。
  - [ ] adapter 失败不能阻断 agent 主流程。
  - [ ] adapter 支持启用、禁用、配置采样、配置脱敏策略。
  - [ ] adapter 输出统一事件给 collector，不直接耦合前端看板。

#### FEAT-03: OpenClaw 首批适配方案

- **描述**: 基于 OpenClaw 现有 plugin hooks 与 diagnostics 能力，设计 OpenClaw adapter。
- **验收标准**:
  - [ ] 明确可通过现有插件/hook 获取的事件类型。
  - [ ] 明确 skill 加载/使用事件是否需要补充通用事件。
  - [ ] 明确不建议通过日志解析作为主路径。
  - [ ] 给出最小侵入改造建议，且改造应面向通用扩展点而非平台定制逻辑。

#### FEAT-04: Hermes Agent 首批适配方案

- **描述**: 基于 Hermes Agent 成熟插件系统和 observability 先例，设计 Hermes adapter。
- **验收标准**:
  - [ ] 映射 Hermes 的 session、gateway、LLM、API request、tool、approval、subagent hooks。
  - [ ] 明确 Hermes skill 采集路径：现有 `skill_view` / slash / preload / `.usage.json` 可作为短期信号，长期建议通用 skill lifecycle hook。
  - [ ] 遵循 Hermes 的显式 opt-in 与 standalone plugin 原则。
  - [ ] 说明哪些数据可实时通过 hook 采集，哪些只能作为本地历史补数。

#### FEAT-05: agent 接入评估机制

- **描述**: 在接入任何新 agent 之前，先按统一清单判断是否能接入、接入成本和事件可信度。
- **验收标准**:
  - [ ] 评估清单覆盖入口消息、用户身份、会话、LLM、tool、skill、错误、权限、隐私、插件机制。
  - [ ] 输出 adapter capability matrix，标注“原生支持 / 可插件支持 / 需轻量改造 / 暂不可支持”。
  - [ ] 评估结果能指导是否进入 adapter 开发。

#### FEAT-06: 统一 collector/exporter 边界

- **描述**: adapter 只负责采集和标准化事件，collector 负责接收、缓冲、校验、入库或转发。
- **验收标准**:
  - [ ] adapter 与 collector 之间有稳定协议边界。
  - [ ] collector 支持后续扩展本地 JSONL、HTTP ingestion、消息队列或数据库写入。
  - [ ] 事件上报失败时有降级策略，不影响 agent 主流程。

#### FEAT-07: 隐私、脱敏与显式启用策略

- **描述**: 遥测能力必须可控、可审计，尤其避免直接外发用户问题、工具参数、文件内容和凭证。
- **验收标准**:
  - [ ] 默认策略不采集 secret，不明文上报凭证、token、cookie、API key。
  - [ ] 原始用户问题与工具参数是否采集必须有显式配置。
  - [ ] 用户 ID、渠道 ID 支持 hash 或映射化。
  - [ ] 每个 adapter 文档必须说明其可见数据范围。

#### FEAT-08: 新 agent 快速接入模板

- **描述**: 为后续 GenericAgent 或其他 agent 预留快速适配路径。
- **验收标准**:
  - [ ] 提供 adapter scaffold 与事件映射模板。
  - [ ] 提供接入检查表与最小示例。
  - [ ] 新 agent 可先以“能力评估”状态进入，不强制完整接入。

#### FEAT-09: 指标与看板延后承载约束

- **描述**: 本期不展开“哪些用户频率高、问了哪些问题、哪些 skill 被调用”等具体指标口径，但事件架构需要能承载这些后续分析。
- **验收标准**:
  - [ ] PRD 和后续设计明确具体指标、报表、前端展示不属于本期 P0。
  - [ ] 事件字段保留足够上下文关联能力，避免后续指标定义时推倒重来。

### 4.2 边缘情况

| 场景 | 预期行为 |
|------|----------|
| 某个 agent 没有插件系统 | 先按接入评估清单标记为“需轻量改造”或“暂不可支持”，不强行接入。 |
| 某个 agent 只能通过日志拿到部分数据 | 日志只能作为补数或 POC 手段，不作为长期主采集架构。 |
| adapter 采集失败或 collector 不可用 | agent 主流程继续执行；adapter 记录失败并按配置降级。 |
| skill 概念在不同 agent 中语义不同 | 事件抽象需区分 `skill`、`tool`、`knowledge_artifact`、`runtime_capability`，避免强行统一为单一字段。 |
| 用户关闭遥测或未配置遥测 | adapter 不采集、不上报；平台应能显示该 agent 未启用遥测。 |
| OpenClaw/Hermes 的上游 hooks 不足 | 优先提出通用 hook 扩展建议，避免为 muad-openclaw 写运行时特例。 |

## 5. 非功能需求

| 指标ID | 类型 | 要求 |
|--------|------|------|
| NFR-PERF-01 | 性能 | adapter 采集逻辑不得显著增加 agent 单轮执行延迟；耗时操作应异步、缓冲或降级处理，具体阈值设计阶段再定。 |
| NFR-REL-01 | 可用性 | 遥测链路失败不得影响 agent 正常对话、工具调用和任务执行。 |
| NFR-SEC-01 | 安全 | 所有 adapter 默认遵循最小采集原则，secret 与凭证类内容必须脱敏或禁止采集。 |
| NFR-SEC-02 | 安全 | 外发遥测必须显式启用，且采集范围可配置、可审计。 |
| NFR-COMPAT-01 | 兼容性 | 事件协议需要支持多 agent、多 runtime、多版本 adapter 共存。 |
| NFR-COMPAT-02 | 兼容性 | adapter 设计应优先使用 agent 官方插件/hook 扩展点，减少对 agent 内部实现的脆弱依赖。 |

## 6. 范围与边界

| 类别 | 内容 |
|------|------|
| **范围（In Scope）** | 统一事件抽象；插件化 adapter 架构；OpenClaw adapter 方案细化；Hermes Agent adapter 方案细化；agent 接入评估清单；collector/exporter 职责边界；隐私与 opt-in 原则。 |
| **非范围（Out of Scope）** | 本期不定义最终业务指标口径；不做完整前端数据看板；不承诺适配所有 agent；不把 GenericAgent 作为首批实现对象；不通过日志解析作为长期主方案；不直接改 OpenClaw/Hermes 核心业务逻辑实现平台特例。 |
| **前置假设** | OpenClaw 与 Hermes Agent 均允许通过插件或通用 hook 接入；muad-openclaw 平台可新增遥测接收与存储能力；具体指标需求会在架构方案确认后另行细化。 |

## 7. 依赖与风险

### 7.1 项目依赖

| 依赖方 | 依赖内容 | 最晚交付时间 | 风险等级 |
|--------|----------|--------------|---------|
| OpenClaw runtime | 插件/hook/diagnostic 事件能力与 skill 生命周期可见性 | 设计阶段确认 | 中 |
| Hermes Agent runtime | 插件启用方式、hooks payload、observability 约束、skill 使用语义 | 设计阶段确认 | 中 |
| muad-openclaw 后端 | 遥测 ingestion、存储、权限与配置能力 | 实施阶段确认 | 中 |
| 用户/产品侧 | 具体指标、报表、保留周期、敏感内容采集边界 | 后续需求阶段确认 | 高 |

### 7.2 风险识别

| 风险ID | 描述 | 影响 | 缓解方案 |
|--------|------|------|---------|
| RISK-01 | 不同 agent 的 skill/tool/message 概念不一致 | 统一事件模型设计不当会导致后续指标失真 | 在事件模型中保留 runtime 原始语义字段，并通过 adapter 做能力声明。 |
| RISK-02 | adapter 过度依赖 agent 内部实现 | agent 升级后埋点失效 | 优先使用官方插件/hook；必要改造只增加通用扩展点。 |
| RISK-03 | 用户内容或 secret 被过度采集 | 产生安全与合规风险 | 默认关闭外发，配置化采集范围，统一脱敏与审计。 |
| RISK-04 | 先做看板指标导致架构过早固化 | 后续新增 agent 或指标时需要重构 | 本期只锁定事件与接入抽象，指标口径后置。 |
| RISK-05 | OpenClaw 或 Hermes 某些关键事件不可见 | 首批适配能力不完整 | 在 capability matrix 标记缺口，并提出最小通用 hook 扩展建议。 |

## 附录：术语表

| 术语 | 定义 |
|------|------|
| Agent Runtime | 具体 agent 底座或运行时，例如 OpenClaw、Hermes Agent、GenericAgent。 |
| Adapter | 面向某个 agent runtime 的遥测适配插件，负责采集原始事件并转换为统一事件。 |
| Collector | 平台侧遥测接收组件，负责接收、校验、缓冲、存储或转发统一事件。 |
| Exporter | 将统一事件输出到具体目的地的组件，例如本地文件、HTTP 服务、数据库、外部观测系统。 |
| Hook | agent runtime 暴露的生命周期扩展点。 |
| Skill | agent 中可复用能力单元。不同 runtime 语义不同，需在 adapter 中明确映射。 |
| Knowledge Artifact | SOP、记忆文件、参考资料等不一定是正式 skill 但会影响 agent 行为的知识对象。 |
| Opt-in | 显式启用。未启用时不采集或不上报遥测数据。 |
