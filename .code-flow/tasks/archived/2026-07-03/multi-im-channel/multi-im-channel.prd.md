## 2. 背景与目标

### 2.1 问题陈述 [必填]

| 维度 | 内容 |
|------|------|
| **问题描述** | 当前 muad-openclaw 每个 Docker 容器仅支持单一 IM 通道（企业微信或微信二选一）。用户若想同时在企微和微信上触达同一个 AI Agent，必须创建两个独立容器，导致会话记忆分裂、资源翻倍、运维复杂度上升。 |
| **当前替代方案** | 为同一用户创建多个容器（如 `alice-wecom`、`alice-wechat`），每个跑不同通道。缺陷：(1) 两个容器的 LLM 会话不共享，用户体验割裂；(2) CPU/内存占用翻倍；(3) 管理员需要维护多份配置。 |
| **触发原因** | 用户实际使用中需要多 IM 同时在线（如企业微信办公 + 微信生活场景），且未来将接入 Slack、Telegram 等更多通道。现有单通道架构限制了平台扩展性。 |

### 2.2 目标与价值 [必填]

| 维度 | 内容 |
|------|------|
| **核心目标** | 一个容器支持配置多个 IM 通道，管理员可在控制台对通道进行新增、更新凭证、删减，变更通过热更新即时生效，无需重启容器。 |
| **预期价值** | (1) 统一会话体验——多平台消息到达同一 AI，共享记忆；(2) 降低资源成本——N 个通道共享一个容器的 CPU/内存；(3) 简化运维——一个用户一个实体，而非 N 个；(4) 为后续接入更多 IM（Slack、Telegram 等）奠定可扩展基础。 |

**成功指标**

| 指标 | 当前值 | 目标值 | 衡量方式 |
|------|--------|--------|----------|
| 单用户支持 IM 通道数 | 1 | ≥2（可扩展至 N） | 控制台创建/编辑可勾选多个通道 |
| 通道变更生效方式 | 容器重启 | 热更新（~200ms） | 编辑保存后容器状态保持 running，通道即刻生效 |
| 容器列表操作按钮数 | 6 + 下拉 | ≤4 可见按钮 | 每行可见按钮 ≤4，批量操作上移表头 |
| 新增 IM 通道类型的改动范围 | 全链路改动 | 仅新增数据定义 | 加一个新通道只改通道注册表 |

#### FEAT-01: 多通道数据模型

- **描述**: 将 users 表的单值 `channel` + `bot_id` + `secret_enc` 列重构为 `channels` JSON 数组和 `channel_configs` JSON 凭证对象，支持每个用户配置 1~N 个通道。
- **验收标准**:
  - [ ] `users.channels` 存储 JSON 数组如 `["wecom","wechat"]`
  - [ ] `users.channel_configs` 存储 JSON 对象如 `{"wecom":{"botId":"...","secret":"..."},"wechat":{}}`
  - [ ] 启动时自动迁移旧单通道数据，botId/secret 归入对应通道配置
  - [ ] `UserSpec` 结构体同步更新，`Channel string` → `Channels []string` + `ChannelConfigs map[string]json.RawMessage`
  - [ ] API 创建/编辑接口接受新字段格式

#### FEAT-02: 数据驱动通道注册表

- **描述**: 前端 `channels.ts` 从纯展示数组扩展为包含凭证字段声明的注册表，表单和展示组件通过遍历注册表渲染，新增通道只需追加数据。
- **验收标准**:
  - [ ] `ChannelDef` 类型包含: `id`, `label`, `icon`, `credentialFields: {key, label, type, required, placeholder}[]`, `hint?`
  - [ ] 企业微信声明 `botId`(text, required) + `secret`(password, required)
  - [ ] 微信声明零凭证字段 + hint "无需凭证，创建后扫码登录"
  - [ ] 未来加 Slack 只需在数组中加一条，不改任何组件逻辑

#### FEAT-03: 多渠道创建表单

- **描述**: 创建容器弹窗中，通道选择从单选 Select 改为多选 Checkbox 组，勾选的通道在下方内联展开凭证输入区。
- **验收标准**:
  - [ ] 所有可用通道以 Checkbox + 图标 + 标签形式列出
  - [ ] 勾选通道后，其凭证字段（按 `credentialFields` 声明）在下方内联展开
  - [ ] 免凭证通道（如微信）展开后显示 hint 文本，不显示输入框
  - [ ] 客户端校验：已勾选通道的 required 字段必填
  - [ ] 未勾选通道不参与校验
  - [ ] 至少勾选一个通道才可提交

#### FEAT-04: 通道编辑表单

- **描述**: 编辑弹窗复用通道选择组件，预填已有配置状态，统一处理追加/删减/更新三种操作。
- **验收标准**:
  - [ ] 已有通道默认勾选，凭证敏感字段（secret）不回填（显示"已配置 · 上次更新于 XX"占位）
  - [ ] 非敏感字段（botId）回填明文
  - [ ] secret 输入框留空 = 保持原值不更新
  - [ ] 取消勾选已有通道 → 提交时清除该通道配置及凭证
  - [ ] 新勾选未配置通道 → 凭证字段为空，required 校验生效
  - [ ] 保存时计算 diff，仅变更的通道触发更新
  - [ ] 取消最后一个通道时给出二次确认："所有通道已取消，用户将无法通过 IM 访问。确认？"

#### FEAT-05: 热更新通道配置

- **描述**: 通道变更通过 `RuntimeDriver.Exec` 接口修改容器内 `openclaw.json`，利用上游 Gateway 的配置 watch + hybrid 热重载模式，通道变更 ~200ms 生效，无需容器重启。
- **验收标准**:
  - [ ] 后端构建完整的 `channels` 配置 JSON（所有已选通道）
  - [ ] 通过 `Exec` 在容器内执行脚本：读取 `openclaw.json` → 更新 `channels` 段 → 写回
  - [ ] Gateway 自动检测文件变更并热应用（`gateway.reload.mode: hybrid`）
  - [ ] 变更后容器状态保持 `running`，不触发重启
  - [ ] Docker driver 和 K8s driver 通过同一 `Exec` 接口工作，无运行时差异
  - [ ] 如果 Gateway 未运行或热重载失败，写入的配置在下次启动时生效（最终一致性兜底）

#### FEAT-06: 容器列表多通道展示

- **描述**: 列表"消息通道"列展示多个 Tag，每通道独立显示连接状态（🟢/🔴）。
- **验收标准**:
  - [ ] 每行"消息通道"列渲染 N 个 Tag，格式 `🏢 企微 🟢`
  - [ ] 每个通道的连接状态由 probe 返回的 per-channel 数据驱动
  - [ ] 通道筛选器改为多选："筛选包含通道 ___ 的用户"
  - [ ] 列表默认按 userId 排序（保持现有行为）

#### FEAT-07: 操作按钮重组

- **描述**: 表头增加复选框 + 批量操作按钮；行内精简按钮数量。
- **验收标准**:
  - [ ] 表头工具栏新增：`[全选 ☐] [重载 Skill] [批量升级] [批量删除]`
  - [ ] 每行首列增加复选框
  - [ ] 批量操作按钮在未勾选时置灰，勾选后可用
  - [ ] 行内按钮从 6 个 + 下拉精简为：`[日志] [扫码(仅微信)] [资源] [更多▾(操作下拉)]`
  - [ ] "升级"和"删除"从行内移除，通过表头批量操作执行（单行 = 勾选一行）
  - [ ] 批量删除给出确认弹窗，列明受影响的用户 ID

#### FEAT-08: 旧数据兼容迁移

- **描述**: 系统启动时检测旧格式数据并自动迁移，保证无缝升级。
- **验收标准**:
  - [ ] 启动时扫描 `channel` 列为旧格式（非 JSON 数组）的记录
  - [ ] 自动将 `channel='wecom'` + `bot_id` + `secret_enc` → `channels=["wecom"]` + `channel_configs={"wecom":{"botId":"...","secret":"..."}}`
  - [ ] 迁移后记录迁移日志
  - [ ] 旧列保留（不 DROP），标记为 deprecated，下一大版本清理

### 4.2 边缘情况 [可选]

| 场景 | 预期行为 |
|------|----------|
| 用户未勾选任何通道就提交 | 前端拦截，提示"至少选择一个通道" |
| 编辑时取消所有通道 | 二次确认弹窗，确认后容器变为无通道状态 |
| 企微通道未填 botId 或 secret | 前端校验拦截，提示"企业微信：Bot ID 必填" |
| 更新凭证时只改 botId 不改 secret | secret 留空提交 → 后端保持原 secret 不变 |
| 热更新时 Gateway 恰好未运行 | 写入 openclaw.json 成功，Gateway 下次启动时加载新配置（最终一致性） |
| 同一通道重复勾选 | 前端禁止（Checkbox 天然互斥）；后端去重 |
| 容器内 `openclaw.json` 被手动修改过 | 热更新写入时覆盖 channels 段，其他段保持用户修改 |
| 多个管理员同时编辑同一用户通道 | 后提交者覆盖先提交者（Last Write Wins），无锁机制 |
| K8s 环境下 Pod 有多个容器 | Exec 默认进入第一个名为 `openclaw` 的容器 |
| 微信通道从未扫码登录就移除 | 直接清除配置，无副作用 |

## 5. 非功能需求 [可选]

| 指标ID | 类型 | 要求 |
|--------|------|------|
| NFR-PERF-01 | 性能 | 热更新通道配置生效时间 ≤ 2s（上游 ~200ms + Exec 调用开销） |
| NFR-REL-01 | 可用性 | 热更新失败不阻塞 API 响应——写入配置后返回成功，Gateway 异步加载；失败时告警但不影响容器运行状态 |
| NFR-SEC-01 | 安全 | 凭证（secret）在 API 传输中使用 TLS（生产环境）；DB 中 `channel_configs.secret` 经 AES-GCM 加密存储（沿用现有 cipher 机制）；热更新写入 `openclaw.json` 时不经过 shell 参数传递（避免 `ps aux` 泄露），改用 stdin 管道或临时文件重定向 |
| NFR-COMPAT-01 | 兼容性 | Docker driver 和 K8s driver 均支持热更新；旧单通道格式数据自动迁移；上游 OpenClaw gateway 的 `hybrid` 热重载模式已默认启用，无需额外配置 |
| NFR-COMPAT-02 | 兼容性 | 支持 Docker 和 K8s（Kubernetes）两种 runtime，热更新通过 `Exec` 接口统一实现 |

## 6. 范围与边界 [必填]

| 类别 | 内容 |
|------|------|
| **范围（In Scope）** | ① 一个容器配置多个 IM 通道（企业微信 + 微信，架构可扩展到 Slack/Telegram 等）；② 通道的增/删/改操作通过统一编辑表单完成；③ 凭证敏感字段留空保持原值、非敏感字段明文回填；④ 通过修改容器内 `openclaw.json` 实现热更新，不重启容器；⑤ 数据模型从单通道列迁移到 JSON 多通道格式；⑥ 容器列表多通道 Tag 展示 + 独立连接状态；⑦ 表头批量操作（升级/删除/重载 Skill）+ 行内按钮精简；⑧ Docker + K8s 同步支持 |
| **非范围（Out of Scope）** | ① 同一平台多账号（如同一用户两个企微 bot）；② 新增 Slack/Telegram/DingTalk 等具体通道的插件安装和通道注册（本次只做架构可扩展性，实际新通道在后续需求中追加数据定义）；③ 热更新的事务回滚（写入失败时不做自动回滚，靠最终一致性兜底）；④ 通道级别的会话隔离（所有通道共享同一 LLM 会话）；⑤ 终端用户自助绑定 IM（仅管理员操作）；⑥ 通道使用量统计/计费 |
| **前置假设** | ① 上游 OpenClaw gateway 默认 `reload.mode: hybrid`，通道配置变更可热加载（已通过文档核实）；② 容器内 `openclaw.json` 的 `channels` 段由 muad 管理，用户不手动编辑；③ `RuntimeDriver.Exec` 在 Docker 和 K8s 均可正常工作；④ 管理员理解多通道概念，知道每个通道需要独立凭证 |

### 7.2 风险识别

| 风险ID | 描述 | 影响 | 缓解方案 |
|--------|------|------|---------|
| RISK-01 | 上游 Gateway 配置热重载在特定场景下不生效（如插件版本不兼容） | 通道变更需重启容器才能生效 | 监控变更后通道状态；若 30s 内未生效则告警提示管理员手动重启 |
| RISK-02 | 直接修改 `openclaw.json` 可能因 JSON 格式错误导致 Gateway 启动失败（下次重启时） | 容器下次启动异常 | 写入前校验 JSON 合法性；写入后保留备份文件（`.bak`）；Exec 返回时验证写入内容 |
| RISK-03 | 多个管理员并发编辑同一用户通道 | 后者覆盖前者 | Phase 1 不做锁，Last Write Wins；文档提示"避免多人同时编辑同一用户" |
| RISK-04 | 热更新写入 credentials 时敏感信息通过 `docker exec` 参数传递暴露在宿主机 `ps aux` | secret 泄露 | 通过 stdin 传递脚本内容，不在命令行参数中携带凭证 |
